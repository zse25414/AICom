# Lumina AI — 知識庫 UX 缺口報告（只分析，無 code）

> 範圍：`lumina-ai.html` 團隊區、教練 RAG 選庫、來源呈現、相關 JS 行為。對照 `TeamPlan.md` 目標能力，標出「已有半成品」與「體驗債」。

---

## 1. 知識庫相關 UX 現況

### 1.1 團隊文件上傳（團隊區 + OCR 提示）

| 面向 | 現況 | 體驗評價 |
|------|------|----------|
| **入口** | 掛在「團隊模式」→「團隊知識庫與新人指南」面板；**非獨立導航** | 發現成本高；成員常以為「教練有知識」但不知在團隊區管理 |
| **權限** | 僅主管可見「新增文件」；成員只讀列表 | 合理，但成員缺「如何請主管補資料」的引導 CTA |
| **類型** | 純文字筆記 / 檔案（PDF、圖片、Excel） | 與產品願景（Word/MD/程式碼）有落差 |
| **大小** | 硬上限 **5MB**（toast 擋） | 有擋、但表單未在選檔前強調「掃描 PDF 常超限」 |
| **解析回饋** | 選檔後「載入並解析…」→ 字元數 / 圖片預覽 | 本機解析有回饋；**發布→雲端／RAG 索引**幾乎無進度 |
| **OCR** | 明確寫「暫不支援自動 OCR」；圖片**強制手填描述**才可發 | 誠實，但「圖片已載入」像成功、實際仍缺可索引文字，易誤判 |
| **類別** | 下拉 4 個固定 `kbId`：general / onboarding / specs / meetings | 是「標籤分類」，不是可管理的多知識庫 |
| **列表** | 類型 badge + 知識庫 badge、PDF/Excel 外開、文字預覽 500 字 | 可用；**無「RAG 已索引／失敗」狀態** |
| **空狀態** | 「目前沒有知識庫文件」+ 主管／成員不同文案 | 合格，但未導向「加入團隊／開 RAG」 |
| **刪除** | confirm 後刪企業文件 + 盡力刪 RAG 索引 | 主管僅 trash icon，缺可讀標籤 |

**雙寫路徑隱債**：發布成功 toast「文件已成功發布」後，另呼叫 `syncDocumentToRag`；RAG 失敗才另 toast。使用者心智裡「已發布」≠「教練查得到」，UI 未把兩階段狀態分開。

**Wire 建議（上傳卡片）**

```
[ 發布新文件 ]
類型: ( ) 文字  ( ) 檔案
標題: ________
知識庫: [ 新人培訓 ▾ ]  [管理知識庫…]
檔案: [ 選擇或拖放 ]  支援 PDF·Excel·圖片 · 最大 5MB · 圖片需手寫說明（無 OCR）
狀態列: ① 本機解析 OK (2,340 字)  →  ② 上傳中…  →  ③ 已索引／索引失敗 [重試]
```

### 1.2 RAG KB 多選（`rag-kb-selector` / checkbox）

| 面向 | 現況 | 體驗評價 |
|------|------|----------|
| **位置** | 僅教練對話 composer 上方 | 正確「消費點」；但**管理點**不在同一心智地圖 |
| **顯隱** | 需 `enterpriseSession` + RAG `/health` 通過才顯示；斷線則 **整塊 hidden** | 離線時「消失」而非「不可用說明」→ 使用者以為功能不存在 |
| **選項** | 後端 `kb/list` 有則用，否則 fallback 四個固定 label | 空庫仍顯示四類，**像有庫可查其實可能 0 文件** |
| **預設** | 無勾選時強制勾第一個 | 無法刻意「不用 RAG、只當一般教練」 |
| **互動** | 多選 checkbox，`10px` 字 + `w-3 h-3` | 資訊密度高、觸控／視力不友善 |
| **持久化** | session 狀態；重整後依健康檢查重繪 | 無「上次選用」明確記憶敘事 |

**與 readiness bar 的關係**：教練有「知識庫教練就緒檢查」（登入／團隊／RAG／已選庫／AI），但 RAG／已選庫 **missing 時不可點、無 action**，修復路徑斷裂。

### 1.3 教練回答 sources 呈現

| 面向 | 現況 | 體驗評價 |
|------|------|----------|
| **何時有** | 僅 `ragServiceActive` + 有勾選 KB + RAG query **HTTP OK** | RAG 失敗時**靜默降級**一般 AI，不標「未使用知識庫」 |
| **UI** | 「資料來源引用」+ chip：`[ref]` + KB 名 + filename + `(xx%)` | 有引用感，但 **span 不可點** |
| **可信度** | `Math.round(score * 100)%` | score 來自融合檢索分數，**不是「答案正確機率」**；標 % 易誤導 |
| **內容** | 無 snippet、無頁碼、無對應文件連結 | 無法驗真，企業場景可信度不足 |
| **與正文** | 後端要求回答內 `[1]`；前端不解析內文標註與 chip 聯動 | 正文編號與底部 chip 未互動高亮 |
| **備援路徑** | 非 RAG 時仍可能把前 10 份文件全文塞進 system prompt | 兩套知識注入邏輯並存，**UI 無法說明「這次是全文注入還是檢索」** |

**Wire 建議（來源 chip）**

```
資料來源
[1] 新人培訓 · 環境架設.pdf          相關度 高
    「…請先安裝 Node 18…」          [開啟文件] [展開片段]
[2] 一般預設 · SOP.md                相關度 中
    （此則為文字筆記 · 可展開）
提示：相關度反映檢索匹配，非答案正確率
```

## 2. 缺失流程盤點

### 2.1 建立／切換／管理多知識庫

| 能力 | 狀態 | 缺口 |
|------|------|------|
| 固定 4 類標籤上傳 | ✅ 半成品 | 無法新建「專案 A／法遵」等真實多庫 |
| 列表依 `kbId` 篩選／分頁 | ❌ | 文件一多只能掃長列表 |
| 重新命名／封存／刪除庫 | ❌ | 後端有目錄級 KB，前端無生命週期 UI |
| 預設查詢庫（團隊預設） | ❌ | 每人各自 checkbox，無團隊共識預設 |
| 庫內文件數／最後更新 | ❌ | 選庫時無「空庫」警告 |

### 2.2 聊天選庫／@KB

| 能力 | 狀態 | 缺口 |
|------|------|------|
| Composer 多選 checkbox | ✅ | 僅此一路徑 |
| 對話中 `@新人培訓` 切庫 | ❌ | TeamPlan 明列未做 |
| 單則訊息覆寫庫選擇 | ❌ | 無法「這題只查會議 SOP」 |
| 訊息氣泡標註「依 X,Y 庫回答」 | ❌ | 事後難追溯 |

### 2.3 來源可點開、可信度、空狀態／RAG 離線

| 場景 | 現況 | 應有體驗（wire） |
|------|------|------------------|
| 來源點開 | 不可點 | 開預覽 drawer：snippet + 回文件列表定位 |
| 可信度 | 偽 % | 改「高／中／低」或分數條 + 說明 |
| 無命中 | 後端文案可能「無法回答」，前端仍可能無 sources | 明確 empty：「知識庫無相關段落」+ 建議換庫／上傳 |
| RAG 離線 | selector 隱藏；設定頁有服務狀態點 | 教練區 inline banner：「RAG 離線，改用一般教練」+ 連到設定／重試 |
| 未加入團隊 | readiness 有 team chip | 可再加「知識庫需團隊」一句說明 |

### 2.4 上傳進度、失敗、檔案類型限制說明

| 階段 | 現況 | 缺口 |
|------|------|------|
| 選檔類型錯誤 | toast「僅支援 PDF…」 | OK，但 Word/MD 使用者會撞牆且無替代建議 |
| 解析中 | 文字「載入並解析」 | 無 cancel、無逾時文案 |
| PDF 掃圖 0 字 | 仍可發布、字元 0 | **未攔截「幾乎無文字」**，RAG 會空 |
| 發布／索引 | 成功／失敗 toast | 無 progress、無文件列「索引中／失敗」徽章、無重試按鈕 |
| 離線團隊 | 本機 store + 仍嘗試 RAG | 離線語意與索引狀態易混亂 |

## 3. 可用性／a11y 風險

### 3.1 大頁面 `lumina-ai.html`（約 1.4k+ 行、全站 SPA）

- **認知負載**：今日／任務／教練／更多（指南·洞察·**團隊**·設定）— 知識庫是「更多 → 團隊 → 頁內下半」三層深。
- **狀態分散**：設定「RAG 連線」、教練 readiness、composer 選庫、團隊上傳 — **四個表面講同一件事**，無單一「知識庫健康」故事。
- **動態 DOM**：列表／thread 大量 `innerHTML`，螢幕閱讀器難追「剛新增的來源／文件」。

### 3.2 觸控

| 風險 | 位置 | 嚴重度 |
|------|------|--------|
| Checkbox 約 12px | `rag-kb` | 高 |
| 刪除僅 icon | 文件卡 | 中 |
| 底部 mobile nav 無「團隊／知識庫」 | 需「更多」sheet | 中（發現性） |
| 教練 chat + 選庫 + quick actions 擠在 composer | 小螢幕易誤觸、遮鍵盤 | 中 |

### 3.3 鍵盤／a11y

| 風險 | 說明 |
|------|------|
| 來源 chip 非 interactive | 鍵盤無法「開啟來源」 |
| 刪除 button 缺 `aria-label` | 僅 icon |
| readiness missing 且 disabled | 無「如何修復」的可聚焦鏈路（RAG 項） |
| 表單 label 多為視覺 label | 多數有關聯；file input 原生可接受 |
| 字級 9–10px badge／checkbox | 對比尚可、**可讀尺寸不足** |
| 跳過連結、`aria-live` toast 已有 | 基礎可；知識庫流程未吃到 |

### 3.4 信任與安全心智（UX 層）

- 成員看到文件列表，但**不知自己問教練時實際查了哪些庫**。
- 主管刪除後文案提「AI 也無法讀取」，但刪 RAG 失敗時**無 UI 回報**。

## 4. 建議資訊架構

### 4.1 是否應獨立「知識庫」？

**建議：邏輯上獨立、導航上「附屬於團隊、可深連」— 不必一開始搶主導航四格。**

| 區塊 | 角色 | 與知識庫關係 |
|------|------|----------------|
| **個人**（今日／任務／設定） | 個人生產力 | 不擁有團隊庫；可顯示「未加入團隊則無 KB」 |
| **教練** | 消費與問答 | **選庫、來源、RAG 狀態**的主要使用面 |
| **團隊** | 成員、任務、權限 | 入口「知識庫」卡片進子頁／子面板 |
| **知識庫**（子目的地） | 上傳、分類、索引狀態、空庫管理 | **管理面**；不應塞進教練 composer |

**為何暫不建議主導航第五項「知識庫」**

- 未加入企業時空白率高，會污染個人路徑。
- 產品核心仍是「今日第一步 + 教練」；KB 是團隊加值。

**建議導航 wire（加入團隊後）**

```
Desktop:  今日 | 任務 | 教練 | 更多 ▾
                    更多: 團隊 | 知識庫 | 洞察 | 設定 | 指南
                         （或 團隊頁內 tab: 總覽 | 任務 | 知識庫）

Mobile:   今日 | 任務 | 教練 | 更多
          更多 sheet 增加「知識庫」捷徑（有未索引失敗時小紅點）
```

**替代（更省導航槽）**：`團隊` 內 **Tabs：成員與任務 | 知識庫**。知識庫從「頁面底部 panel」升成 **一級 tab**，成本低、IA 清晰。

### 4.2 心智模型一句話

> **團隊頁管理「庫與文件」；教練頁決定「這次問哪些庫」；設定頁只負責「服務是否活著」。**

## 5. P1 最小 UX 改動清單（最多 5，影響力排序）

僅 UI／文案／流程可見性；不擴後端能力也可先做前幾項。

| 順位 | 改動 | 解決什麼 | 影響面 | 粗估工量* |
|------|------|----------|--------|-----------|
| **P1-1** | **RAG 狀態可見**：離線時不要整塊隱藏選庫；改 disabled + 一句說明；query 失敗／降級時在訊息下標「未使用知識庫」 | 信任、除錯、避免「以為有 RAG」 | 教練 | S |
| **P1-2** | **來源可理解 + 可點**：去掉誤導性「%」改相關度等級；chip 可開 snippet／連到團隊文件（有 fileUrl 則開檔） | 可追溯、企業可信 | 教練 sources | S–M |
| **P1-3** | **上傳雙階段狀態**：文件卡徽章「已發布／索引中／索引失敗」+ 失敗重試；「幾乎 0 字」PDF 警告 | 減少「發了但教練答不出」 | 團隊知識庫 | M |
| **P1-4** | **選庫可用性**：加大 hit area、顯示各庫文件數或「空」；允許「不勾選＝純教練」；composer 一句「目前查：A,B」 | 正確使用多庫、觸控 | 教練 selector | S |
| **P1-5** | **IA 降發現成本**：團隊頁「知識庫」升 tab 或更多選單捷徑；readiness 的 RAG chip 連到設定「重新檢查」 | 找得到、修得到 | 導航 + readiness | S |

\*S ≈ 0.5–1 人日文案／前端；M ≈ 1–2 人日（狀態欄位與列表聯動）。

**刻意不進 P1 的（P2+）**

- 自建多知識庫 CRUD、`@KB`、OCR、Word/MD、版本管理 — 產品力大，但依賴 API／解析，不屬「最小 UX」。

## 坑點總表（嚴重度）

| 嚴重度 | 坑點 |
|--------|------|
| **高** | 發布成功 ≠ RAG 索引成功，無持久狀態 |
| **高** | RAG 離線／失敗時選庫消失 + 靜默降級 |
| **高** | 來源分數當「% 可信度」誤導 |
| **中** | 知識庫深埋團隊頁，與教練消費面脫節 |
| **中** | 固定四類假「多庫」，空庫仍可勾 |
| **中** | 掃圖 PDF／無 OCR 圖片易進空索引 |
| **低–中** | 小字、小 checkbox、刪除 icon a11y |
| **低** | 兩套知識注入（全文 prompt vs RAG）使用者無感 |

## 總體結論與權衡

| 方案 | 優點 | 缺點 | 建議 |
|------|------|------|------|
| **A. 只修教練面（P1-1/2/4）** | 最快提升信任 | 管理面仍亂 | 若本週只能做一刀：先 A |
| **B. A + 上傳狀態（P1-3）** | 閉環「寫入→可問」 | 需標狀態資料 | **推薦 MVP 閉環** |
| **C. 獨立知識庫大頁 + 自建庫** | 對齊 TeamPlan | 工期與導航變動大 | 放進下一階段，先做 tab 升格 |

**推薦路徑**：採 **B** 作為 P1 包；導航用 **團隊內知識庫 tab／更多捷徑**（P1-5）代替立刻做頂層「知識庫」主導航。獨立多庫與 `@KB` 等能力等 P1 信任閉環穩定後再做，避免「功能變多但更不可信」。
